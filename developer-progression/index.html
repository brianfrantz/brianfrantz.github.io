Developer Progression

29 May, 2014

I've been thinking a lot recently about the progression of a developer from just starting out through to "expert" status. I'm not sure if other folks really think about this, but I'm finding that where a developer is in their progression can help predict their successes and failures.

Here is my view of the typical progression of a software developer:

  • Brand New Developer - At first, our brand new developer is intimidated by what to type. The syntax. Figuring out how to debug that missing semi-colon. All thought goes to figuring out the syntax - very little time go towards understanding the problem to be solved. Focus: Syntax.

  • Features (Kinda) Work - After a while our developer is able to build a solution to a given problem that works in the ideal case. When put in front of an actual user (or hopefully a tester first) things immediately start to fall apart. Entering 21 characters on a field where the database column is set to 20 characters causes an app crash. That kind of thing. Focus: Code.

  • Features Actually Work - Assuming our developer survives the beatings administered by irate testers and/or users, those lessons result in complete features from the user's point of view. From a senior developer's point of view though, our developer's code just isn't where it needs to be. It probably features lots of duplication, deeply nested if statements, dead code, over engineering, gold plating, and other scary stuff. Unfortunately a lot of developers max out at this level in the progression - they are satisfied with good-enough solutions and rough edges. The problem is "solved" and they move onto the next one. Focus: Correctness.

  • Writes Pretty Code - The code smells and resulting code maintenance pain eventually drive our developer to start refactoring his/her code. Our developer starts taking pride in writing high quality code that he/she can be proud of. Not only does the feature work, but the code is as simple as it can be and is understandable. Many times developers at this level are zealots for their language of choice - they have worked hard to really understand its ins and outs. It is their primary tool to solve any problem, even if it isn't necessarily the best tool for the job. Focus: Tight, well written, complete code

  • Considers the "ilities" - Code can be well written but still be deficient. New terms start to enter our developer's vocabulary: the "ilities". Scalability, usability, maintainability, reliability, extensibility, security (securit-ility?). Our developer is now spending a lot of time considering how the feature fits into the overall system. At this point the programming language becomes secondary to the solution - with experience in multiple languages and technologies comes knowledge of the strengths and weaknesses of each. With a full tool belt our developer can now build better solutions. Focus: Complete Solutions

  • Writes Simplest Possible Code - This is the top of the progression. Given all of the above, our developer now has the experience to know how much engineering is appropriate for the given scenario. He/she has a full tool belt of techniques, technologies, and approaches and can apply the best tool for the job. Our developer understands that while some code needs to be armor plated to handle any eventuality, there are scenarios where quick and dirty is the appropriate level of effort. Focus: Best Solutions

Unfortunately not everyone will get to the end of the progression and there is generally a lot of "plateau-ing" along the way. Moving to the next level depends a lot on the environment the developer is in, especially whether or not he/she is working closely with developers that are further along in the progression. Mentoring, either explicit or implicit, is a huge factor in moving along in the progression.

When evaluating a developer, it is important to realize that this progression cannot be bypassed and advancement can only come through hard-won experience. Each level is a prerequisite for the level beyond it and will take significant time to complete. Expecting a junior developer to jump right to writing the simplest possible code is setting him/her up to fail.

I've moved along this progression over the course of my career. For you developers out there, do you find this fits your progression over the years? Let me know what you think...
